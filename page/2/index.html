数据结构学习笔记-线性表

发表于 2016-04-27   |   分类于 数据结构学习笔记

线性表的定义

线性表是零个或多个数据元素的有限序列。

从定义中可以发现以下两个特点：

  • 有序性：元素之间有顺序，第一个元素没有前驱，第二个元素没有后继。
  • 有限性：在计算机处理的数据当中，都是有限的。

线性表从存储结构来分类，可以分为顺序存储结构和链式存储结构。

线性表的抽象数据结构

线性表的顺序存储结构

顺序存储结构指的是用一段地址连续的存储单元一次存储线性表的数据元素。使用一维数组来实现，需要给定最大元素数量。

需要指明的是，线性表的数据长度小于等于线性表的存储所在数组的长度。因为数组的长度通通常是固定的，而数据长度可以动态的改变。

查找

  • 判断线性表是否为空，给定索引是否超过线性表长度，返回异常
  • 按照索引返回数据元素

插入

  • 判断插入位置是否在线性表索引当中
  • 如果数组已满，抛出异常或者动态增加容量
  • 从最后一个位置开始，到插入位置，整体向后移一个单位位置。
  • 插入
  • 长度 +1

删除

  • 删除位置不在索引范围当中
  • 取出删除元素
  • 从最后一个元素到删除位置，这个难题向前移动一个单位位置
  • 表的长度 -1

总结

在顺序存储结构中，查询元素和添加新元素的操作，都可以直接通过索引计算出内存地址，所以时间复杂度为O(1)。而插入和删除操作都需要将一部分元素整体移动，所以在最坏情况下，时间复杂度为O(n)。

这说明，线性表的顺序存储结构适用于多存取，少改动的情况。

线性表的链式存储结构

线性表的链式存储结构通常指的是链表。它的出现很好的解决了顺序存储结构的缺点，更有效率的进行插入和删除。

单链表

单链表的每一个节点都包含数据域和指向下一个节点的指针。

双链表

双向链表的每个节点包含两个指针和一个数据域，这样可以加快查找速度，从而提高插入、删除的效率。

静态链表

静态链表的存在，是为了使不具有指针的编程语言实现链表。

静态链表的本质就是用数组来实现链表，数组中的每个元素都有数据域和下一个元素的数组下标。这样的话，在插入和删除数据元素时，不需要整体移动数组，从而提高效率。

循环链表

循环链表存在的意义在于快速查找尾节点。

循环链表就是把最后一个节点的指针指向头节点。也就是说，头节点被两个指针同时指着。

栈

仅能在表尾插入或删除数据的线性表称为栈(Stack)。

存取遵循LIFO原则，就是后进先出。

栈的抽象数据类型中，只有一个指针用于指向栈顶的数据。

队列

只允许在一端插入、一端删除的线性表称为队列(Queen)

字符串

由零个或多个字符组成的有限序列。

时间复杂度的定义

在进行算法的分析时，算法的总执行时间 T(n) 是关于问题规模 n 的函数。记作 T(n) = O(f(n))。也就是我们常说的大 O 表示法。

在这里我们计算的时间复杂度，是对于最坏情况的考察，因为这代表着系统的鲁棒性有保证。平均情况的时间复杂度是对运算时间的一个期望，并不能表示最坏情况。

时间复杂度的计算

时间复杂度的计算，大体上遵从以下规律，实际分析当中需要具体分析。

  • 用常数 1 取代所有相加的常数
  • 只保留高阶项
  • 把最高阶项的常数置为1

平方阶

嵌套循环是典型的平方阶复杂度，我们应该由内而外的分析。内层循环执行 n 次，外层循环也执行 n 次，所以时间复杂度为 O(n^2）。

下面增加一点难度：

内层循环的执行次数随着 i 的增加而减少，所以执行次数为等差数列：

n + (n - 1) + (n - 2) + (n - 3) ... = (n * (n + 1)) / 2

化简为多项式形式：n^2 / 2 + n/2，所以按照规则，只保留最高阶项，然后再去掉常数 1/2。 所以结果仍然是 O(n^2)。

看到这里可能会感觉奇怪，为什么以上两个算法的打印语句的执行次数明显不一样，但时间复杂度却一样呢？事实上，时间复杂度的目的就是为了描述算法执行时间的数量级与 n 的关系，只是个粗略的估计。并且时间复杂度的估量是在假设 n 足够大的情况下，用极限的概念来理解的。

空间复杂度的计算

一个算法的优劣，不仅要考量时间复杂度，还要考虑对硬件存储空间的要求。

空间复杂度 S(n) = O(f(n))，其中 n 是问题的规模，f(n) 存储空间关于 n 的函数。

空间复杂度主要取决于执行算饭所需的存储空间。

什么是事件传递？

事件传递说白了就是iOS应用程序对用户操作进行逐级、有序处理的过程，这个过程会由UIWindow开始逐级向子视图进行检查。

第一步：捕捉用户操作

当用户触摸屏幕触发一个触摸事件(Touch Event)时，UIKit 会创建一个包含事件信息的 UIEvent 对象。然后把这个UIEvent对象放到到事件队列当中等待处理。对于触摸事件来说，这个 UIEvent对象是一系列触摸操作的集合。

单例的UIApplication对象会从事件队列(Event Queue)的顶部取出这个UIEvent对象。然后，这个对象会首先被送到当前应用的UIWindow对象，这个UIWindow对象会带着需要处理的UIEvent对象进行一个叫做hit-testing的过程。这个过程说白了就是寻找用户点击的位置到底是属于哪个View。

第二步：通过hit-testing寻找触摸目标

hitTest方法首先会掉用pointInside方法来判断触摸的位置是否在被检查的View中，如果是，那么pointInside方法返回True，然后继续使用hitTest方法检查该View的所有子视图。如果触摸位置不在被检查的View中，那么pointInside方法返回false，因而hitTest方法会返回nil。这样的话，不再检查该View的子类。这就意味着，如果要让某个View的事件被触发，必须保证该View在它的父视图当中。

下面使用一个苹果文档中提供的例子来说明：

如果用户触摸View E，首先，检查View A，用户的操作处于View A，所以下一步检查View A的子类View B和View C。显然，触摸点不在View B中，所以开始检查View C中的两个View。最后，View E被判定为被用户点击的视图。

什么是响应链？

响应链是一系列连在一起的响应者对象，开始于第一响应者(First Respondor)，结束于当前应用程序对象(UIApplication)。响应者对象就是能处理事件的对象，响应者对象的基类是UIResponder，所有响应者都是这个类的子类：

如果想要让某一View成为第一响应者，需要满足两个条件：重写继承自UIResponder的canBecomeFirstResponder方法并返回true，并且接收一个来自becomeFirstResponder方法的message。UIKit自动的将UITextfield和UITextView这两个控件设置为当用户点击时自动成为第一响应者，而其他的控件用户就需要自己设置becomeFirstResponder了。

当我们经过上一步的hitTest方法检查出被点击的视图后，如果该视图不能处理该事件，那么就会使用nextResponder方法将事件传递给响应链的下一个响应者对象，直到被处理或者没有下一个响应者。如果连UIApplication都处理不了事件，那么该事件将会被销毁。

当我们使用 UIScrollView 控件时，会出现有显示内容，但是无法滚动的情况。

解决方法

第一步：确定 UIScrollView 对象的 contentSize 属性是否大于父视图

第二步：检查设置 contentSize 的语句在哪个方法中

上面的语句不能放在 viewDidLoad() 方法当里，在 iOS 8 以上可以放在 viewDidAppear() 方法中。而在 iOS 7 ，可以参考iOS7 下 UIScrollView 无法滑动 中提到的方法，放在 viewDidLayoutSubviews() 方法中。

原理

iOS 的view加载过程会执行很多方法，当 view 加载完毕后，会先执行 viewDidLoad() 方法，然后在开启 Auto Layout 的情况下，会根据自动布局再一次计算 Constraint ，在计算完毕后会执行 viewDidLayoutSubviews() 方法。而最后才会执行 viewDidAppear() 方法。所以我们在 view 即将显示的时候设置 scrollView 的 contentSize 属性就可以啦。
